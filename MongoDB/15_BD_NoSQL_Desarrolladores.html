Guía de bases de datos NoSQL para desarrolladores

¿qué son las bases de datos NoSQL y por qué deberían preocuparme por ellas?

La forma más sencilla de entender la definición es que las bases de datos NoSQL no son bases de datos SQL. Las bases de datos SQL relacionales estándar, como el servidor SQL de Oracle o MySQL, tienen una estructura tabular, similar a la que vería en una hoja de cálculo de Excel. Mientras que las bases de datos NoSQL tienen varios tipos de formatos y, por lo tanto, pueden ser más flexibles.

Importante, la flexibilidad NoSQL es su mayor fortaleza pero también su mayor desafió.

¿Como decidir cual usar? Todo se reduce a cuáles son los requisitos de la aplicación.

SQL vs NoSQL, son herramientas y son las más efectivas cuando se usan en una aplicación que funciona bien con sus puntos fuertes.

Una buena regla general es simplemente seguir las pautas dadas por los propios nombres.:
Si tiene datos que dependen en gran medida de las relaciones, entonces probablemente sea mejor utilizar una base de datos SQL relacional.
Si tiene un conjunto de datos no estructurados que necesita flexibilidad, entonces puede ser la mejor opción optar por una base de datos NoSQL.

Esa es realmente la diferencia clave para entender cómo funcionan

Si está interesado en crear aplicaciones con bases de datos NoSQL, es importante comprender que no existe una base de datos NoSQL estándar, sino que existen muchas variaciones diferentes. Hay almacenes de pares clave/valor, almacenes familiares de columnas anchas, bases de datos de documentos y bases de datos de gráficos.

Por ejemplo

Imagine que tiene que crear una aplicación en la que los usuarios puedan definir sus propios campos, como una aplicación de encuestas en la que un usuario puede crear una encuesta con dos campos y otro puede crear una encuesta con cien campos, y necesita tener la capacidad de ser nombrado sobre la marcha. Técnicamente sería posible implementar esto con una base de datos relacional, sin embargo, sería muy complicado. Sin embargo, las bases de datos NoSQL podrían manejar esto fácilmente, permitiendo que los campos se creen dinámicamente y luego cada registro se almacene con sus características únicas..

Imagine que está creando una aplicación de inventario que tiene campos que deben conectarse a otras tablas, como una tabla de usuario para el empleado que ingresa la información, una tabla de proveedores y una tabla de impuestos para calcular la depreciación. Si intentara implementar una base de datos NoSQL para esta opción de inventario, terminaría teniendo que trabajar muy duro para asignar los valores entre los modelos y también haría que la aplicación fuera muy difícil de mantener. En este caso de estudio, una base de datos relacional (rdms) funcionaría perfectamente.
